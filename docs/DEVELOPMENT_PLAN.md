# План разработки: Смета-Стройка AI Калькулятор

## 1. Цель разработки

Довести проект до полноценной рабочей версии для ИЖС: пользователь запускает локальное приложение, создает или переносит дом из PDF на холст, выбирает конструктивы и материалы, загружает прайсы, получает связанную смету и формирует КП.

Поставка должна работать на Windows и macOS.

Целевой клиентский опыт:

1. Клиент получает папку проекта.
2. Мы персонализируем папку под клиента: название, логотип, реквизиты, базовые настройки.
3. Клиент запускает приложение через простую кнопку/скрипт.
4. Данные клиента сохраняются внутри папки `client-data/`.
5. Codex используется как помощник настройки, анализа PDF, прайсов и обновления проекта.

## 2. Главные принципы

- Не строить временную игрушку. Все этапы должны вести к финальной архитектуре.
- Сохранить текущий прототип как UX-референс.
- Не зависеть только от `localStorage`.
- Делать расчеты прозрачными: источник объема, источник цены, формула, статус проверки.
- AI/Codex предлагает и помогает, но пользователь подтверждает результат.
- Расчет по фундаментам и конструктивам - сметный/коммерческий, не замена инженерного проекта.

## 3. Итерация 0. Подготовка проекта

Цель: безопасно подготовить базу разработки.

Задачи:

1. Создать структуру `legacy/prototype/` и перенести туда текущий CDN/Babel-прототип.
2. Создать новую структуру `src/` на React + TypeScript + Vite.
3. Добавить базовые скрипты:
   - `start.bat`;
   - `start.command`;
   - `backup.bat`;
   - `backup.command`.
4. Создать структуру клиентских данных:
   - `client-data/prices`;
   - `client-data/pdfs`;
   - `client-data/projects`;
   - `client-data/exports`;
   - `client-data/company`;
   - `client-data/templates`.
5. Настроить базовые проверки: TypeScript, lint, форматирование.

Критерий готовности:

- проект запускается на Windows и macOS через скрипт;
- старый прототип сохранен;
- новая app-структура готова к переносу модулей;
- `client-data/` не ломает git и может быть пустой в чистой поставке.

## 4. Итерация 1. Локальное хранение и запуск

Цель: сделать основу, где данные клиента реально сохраняются локально.

Задачи:

1. Добавить локальный Node API для работы с файлами.
2. Подключить SQLite или временный файловый JSON-слой с миграцией к SQLite.
3. Реализовать сохранение/загрузку проектов.
4. Реализовать backup `client-data/` в архив.
5. Добавить страницу настроек компании: название, логотип, реквизиты.
6. Добавить `docs/CODEX_FAQ.md` с командами и сценариями для Codex.

Критерий готовности:

- данные сохраняются не в браузере, а в папке клиента;
- backup создается одной командой/кнопкой;
- проект можно перенести вместе с `client-data/`.

## 5. Итерация 2. Доменная модель ИЖС

Цель: создать расчетное ядро, до сложного UI.

Сущности:

- Project.
- Level: фундамент, 1 этаж, 2 этаж, кровля, гараж, баня, терраса.
- CanvasElement: wall, partition, room, opening, window, door, roofSegment, foundationSegment.
- TechnologyPackage.
- PriceSource.
- PriceItem.
- Estimate.
- EstimateLine.
- Proposal.

Задачи:

1. Описать типы данных.
2. Описать базовые формулы объемов:
   - длина;
   - площадь;
   - периметр;
   - объем;
   - количество;
   - вычет проемов.
3. Добавить статусы:
   - рассчитано;
   - требует проверки;
   - ручная правка;
   - исключено из сметы.

Критерий готовности:

- можно создать проект программно и получить черновик сметы из тестовых данных.

## 7. Итерация 4. PDF и AI-кандидаты

Цель: сделать загрузку PDF/картинки и перенос на холст.

Задачи:

1. Загрузка PDF/изображения в `client-data/pdfs`.
2. Просмотр страниц PDF.
3. Классификация страниц:
   - 1 этаж;
   - 2 этаж;
   - фундамент;
   - кровля;
   - фасад;
   - другое.
4. PDF как подложка.
5. Калибровка масштаба.
6. JSON-кандидаты от Codex/AI:
   - стены;
   - комнаты;
   - размеры;
   - окна;
   - двери;
   - подписи.
7. Экран подтверждения кандидатов.

Критерий готовности:

- пользователь загружает один из реальных планов;
- система переносит/предлагает элементы;
- пользователь подтверждает и получает расчетный холст.

## 8. Итерация 5. Фундаментный блок

Цель: сделать фундамент отдельным уровнем и расчетным модулем.

Типы:

- монолитная плита;
- ленточный фундамент;
- лента + плита;
- забивные сваи;
- буронабивные сваи;
- свайно-ростверковый фундамент;
- УШП как расширенный сценарий.

Расчетные элементы:

- земляные работы;
- песчаная/щебеночная подушка;
- геотекстиль;
- гидроизоляция;
- утепление;
- бетон;
- арматура;
- опалубка;
- сваи;
- ростверк;
- дренаж;
- ливневка;
- отмостка.

Подход к арматуре:

- первый рабочий вариант: формулы, шаги, коэффициенты, ручная корректировка;
- визуальная схема армирования как расчетная иллюстрация;
- детальный инженерный расчет несущей способности не обещать как автоматический результат.

Критерий готовности:

- пользователь выбирает тип фундамента;
- фундамент связан с геометрией дома;
- смета получает материалы по фундаменту;
- пользователь видит предупреждения и может править вручную.

## 10. Итерация 7. Прайсы

Задачи:

Критерий готовности:

- удаленный прайс больше не участвует;
- поставщик виден в смете.

## 11. Итерация 8. Смета и КП

Цель: связать холст, прайсы, работы и документ.

Смета:

- материалы по объекту;
- источники объемов;
- источники цен;
- ручные правки;
- предупреждения.

КП:

- выбор работ по конструктивам;
- цена работ за единицу;
- наценки;
- налоги;
- условия оплаты;
- сроки;
- гарантия;
- экспорт PDF/печать;
- отдельные выгрузки: материалы, работы, общий файл.

Критерий готовности:

- из объекта формируется понятная смета;
- из сметы формируется КП;
- результат можно скачать/распечатать.

## 13. Критические риски и решения

| Риск | Решение |
|---|---|
| PDF распознается неточно | Кандидаты, confidence, ручное подтверждение, fallback-обводка |
| Браузер не пишет в папку | Локальный Node API или Tauri |
| Слишком много строительных сценариев | Идти через технологические пакеты и приоритет ИЖС |
| Ошибка в расчете | Формула, источник, статус проверки, ручная корректировка |
| Клиенту сложно запускать | Скрипты Windows/Mac, позже Tauri |
| Прайсы разного формата | Импорт-маппинг и ручное подтверждение колонок |
| Перегруз интерфейса | Пошаговый сценарий: проект -> PDF/холст -> конструктивы -> прайс -> смета -> КП |

## 14. Критерий финального результата

Финальный результат достигнут, когда:

- продукт запускается на Windows и macOS;
- данные клиента хранятся локально;
- можно загрузить/перенести реальные планы ИЖС;
- можно создать одноэтажный и двухэтажный дом;
- можно посчитать фундамент, стены, коробку, кровлю и основные разделы ИЖС;
- можно загрузить прайсы нескольких поставщиков;
- смета пересчитывается от холста и прайсов;
- КП формируется из сметы и выбранных работ;
- есть экспорт/печать;
- есть Codex FAQ и сценарий настройки под клиента.
