# Предразработочный ресерч и риски

## 1. Codex-powered поставка

Идея возможна, но важно правильно выбрать техническую основу.

Обычная web-страница не может свободно работать с папкой проекта: читать/писать прайсы, PDF, SQLite-файл и экспорты напрямую из браузера нельзя без ограничений.

Рабочая схема:

1. Клиент получает папку проекта.
2. Запускает `start.bat`.
3. Скрипт поднимает локальное приложение.
4. Данные клиента лежат в `client-data/`.
5. Приложение работает с ними через локальный Node API или через Tauri.
6. Codex используется для настройки, анализа файлов, помощи с PDF/прайсами и доработки клиента.

Рекомендация:

- Сначала сделать локальную web-версию с Node API.
- Не хранить рабочие данные только в `localStorage`.
- Сразу проектировать слой хранения так, чтобы потом перенести его в Tauri.
- Поддерживать Windows и macOS с первого этапа: `start.bat`, `start.command`, `backup.bat`, `backup.command`.
- Целевой сценарий клиента: двойной клик, приложение открылось, данные сохраняются в папке клиента.

## 2. Паттерны аналогов

По рынку takeoff/estimating-инструментов повторяются одни и те же сильные паттерны:

- загрузка PDF/изображений;
- многостраничные планы;
- калибровка масштаба;
- инструменты длины, площади, количества;
- слои и группы измерений;
- подтверждение AI-кандидатов человеком;
- assemblies/технологические пакеты;
- связь takeoff-объемов с ценами;
- смета и КП без повторного ручного ввода;
- экспорт PDF/Excel/CSV.

Вывод:

Наша ключевая фича не "AI сам все посчитал", а связка:

> PDF/холст -> подтвержденные объемы -> технологические пакеты -> прайсы поставщиков -> смета -> КП.

## 3. ИЖС и фундамент

Для ИЖС фундамент нельзя делать одним универсальным калькулятором. Нужны сценарии:

- монолитная плита;
- ленточный фундамент;
- лента + плита;
- забивные сваи;
- буронабивные сваи;
- свайно-ростверковый фундамент;
- утепленная плита/УШП как расширенный сценарий;
- дренаж, ливневка, отмостка как связанные доп. блоки.

Расчет должен быть параметрическим:

- геометрия из холста;
- высота/толщина;
- глубина;
- шаг свай/арматуры;
- класс бетона;
- запас материала;
- правила округления;
- ручная корректировка.

Критичный риск:

Фундамент зависит от грунтов, нагрузок и проектных решений. Продукт должен позиционировать расчеты как коммерческую прикидку/сметный расчет, а не инженерный расчет несущей способности.

## 4. Стены, перегородки, проемы

Нужны минимум два базовых сценария:

- газоблок;
- деревянный каркас.

Для газоблока:

- толщина стены;
- высота этажа;
- длина/площадь стен;
- вычет проемов;
- перемычки;
- армирование рядов;
- армопояс;
- клей/раствор;
- запас.

Для каркасника:

- стойки;
- обвязка;
- шаг стоек;
- OSB/мембраны;
- утепление;
- пароизоляция;
- крепеж;
- обрешетка.

## 5. Ключевые слабые места проекта

1. **Автораспознавание PDF.**  
   Самый рискованный блок. Нужно делать через кандидатов, confidence, ручное подтверждение и обязательный fallback на обводку.

2. **Точность расчетов.**  
   Без правил округления, запасов и ручной корректировки смета будет вызывать недоверие.

3. **Прайсы из PDF/сайтов.**  
   Структура у поставщиков разная. Нужен импорт-маппинг: колонка, единица, цена, поставщик, категория.

4. **Файловое хранение.**  
   Нельзя рассчитывать, что браузер сам будет управлять папкой клиента. Нужен локальный backend/Tauri.

5. **Слишком большой первый объем.**  
   "Полный ИЖС" большой. Нужно идти итерациями, но каждая итерация должна строить финальное ядро, а не временную заглушку.

6. **Юридическая/инженерная ответственность.**  
   Нужно явно показывать: расчет коммерческий, требует проверки специалиста.

7. **Миграция текущего прототипа.**  
   Есть риск потерять UX. Поэтому старый прототип надо сохранить в `legacy/prototype/`.

8. **Windows/Mac без ручной установки.**  
   Полностью "ничего не устанавливать" проще достигается через Tauri/упакованный runtime. На раннем этапе локальный Node API может требовать Node.js, поэтому нужно либо автоматизировать проверку/установку, либо быстро перейти к desktop-сборке.

9. **Инженерные расчеты.**  
   Нельзя подменять проектировщика. Нужно делать коммерческие/сметные расчеты материалов и работ, а инженерную несущую способность оставлять как предупреждение и область проверки специалистом.
