# Анализ загруженных материалов

## 1. Реальные сметы

Изучены примеры:

- `Смета_Денис_23.04.2026.pdf`
- `Смета_Никита.pdf`
- `Смета_Сергей 2.0.pdf`

Общий формат:

- номер позиции;
- наименование работ и материалов;
- единица измерения;
- количество;
- цена материалов за единицу;
- цена работ за единицу;
- всего материалов;
- всего работ;
- общий итог.

Повторяющиеся разделы:

- фундамент/коммуникации;
- коробка;
- кровля;
- фасад/цоколь;
- террасы/ступени/входная группа;
- теплоизоляция;
- внутренняя отделка;
- инженерные системы/кухня;
- электрика;
- сантехника;
- отопление/вентиляция;
- дополнительные работы.

Вывод для продукта:

Сметная таблица должна изначально поддерживать разделение материалов и работ. Даже если на холсте первым этапом считаем материалы, модель данных не должна терять работы, потому что блок КП будет их добавлять и пересчитывать.

## 2. ТЗ Сметчик

Документ описывает технологию строительства и правила расчета для ИЖС.

Ключевые технологические блоки:

- плита;
- лента + плита;
- забивные ж/б сваи;
- буронабивные ж/б сваи;
- отмостка;
- ливневая канализация;
- дренаж;
- газоблок;
- деревянный каркас;
- металлочерепица;
- клик-фальц;
- водосточка;
- снегозадержание;
- подшив свесов.

Вывод для продукта:

Нужна сущность `Технологический пакет`: не просто строка прайса, а набор входных параметров, формул, материалов, работ, единиц измерения и правил округления.

## 3. Занятия с Петром

Из транскриптов важны следующие требования:

- инструмент создавался под реальную задачу быстрого расчета смет;
- текущий расчет занимает примерно часы на одну смету;
- пользователь не хочет вручную готовить десятки файлов, значит интерфейс должен помогать собирать структуру;
- нужны эталонные сметы и прайсы как база знаний/пример результата;
- важна возможность голосом/текстом объяснять правила, но в продукте это должно стать структурированными настройками;
- пользователь хочет готовый результат, а не обучение ради обучения;
- в будущем возможны контроль объектов, чек-листы и фотоотчеты.

Вывод для продукта:

AI-логика должна быть встроена в понятные рабочие сценарии: загрузить прайс, сопоставить материал, проверить пропуски, сформировать КП. Не нужно делать отдельного "чат-агента" вместо продукта.

## 4. Практический вывод

Следующая разработка должна идти не от экранов, а от ядра:

1. Доменная модель проекта и элементов холста.
2. Модель прайсов.
3. Модель технологических пакетов.
4. Модель сметы.
5. Модель КП.
6. Потом перенос UI на полноценный стек.
