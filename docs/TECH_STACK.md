# Технический стек и архитектурный курс

## 1. Цель стека

Нужен не прототип, а полноценный масштабируемый продукт, который:

- работает как веб-приложение во время разработки;
- может поставляться клиенту как папка проекта для запуска и настройки через Codex;
- может быть упакован как desktop-приложение для компьютера клиента;
- хранит проекты, прайсы, сметы и КП локально;
- поддерживает сложный 2D-холст;
- в будущем расширяется модулями: тендеры, ПТО, контроль объектов.

## 2. Рекомендуемый стек

### Frontend

- React + TypeScript.
- Vite.
- React Router или TanStack Router.
- Zustand для локального UI/state холста.
- TanStack Query для серверных/локальных async-запросов.
- TanStack Table для прайсов, смет и КП-таблиц.

### Холст

Основной вариант: React Konva / Konva.

Причина:

- canvas-подход подходит для стен, сегментов, привязок, drag/drop, resize, слоев и интерактивного выбора;
- React Konva дает декларативный React-подход поверх canvas;
- текущий SVG-прототип можно переносить поэтапно.

Альтернатива:

- SVG оставить для первого этапа миграции, если нужно быстрее стабилизировать текущий код;
- Fabric.js рассмотреть только если Konva не закроет нужные CAD-like сценарии.

### PDF

- PDF.js для отображения PDF-страниц в браузере/canvas.
- На первом этапе: PDF как подложка + масштаб + ручная обводка.
- Позже: извлечение текста/векторных объектов и AI-кандидаты.

### Данные

Для финальной desktop-версии:

- SQLite как локальная база.
- Tauri SQLite plugin или backend-команды Tauri.
- Drizzle ORM или Kysely для типобезопасного доступа к данным.

Для web/dev-режима:

- IndexedDB как временный local-first слой.
- Dexie или PGlite можно рассмотреть, если потребуется сложная локальная синхронизация.

### Desktop-упаковка

- Tauri 2.

Причина:

- можно использовать текущий web-frontend;
- приложение можно отдавать на компьютер клиента;
- меньше вес, чем у Electron;
- есть доступ к файловой системе, локальной БД, импорту/экспорту.

Важно: Tauri закладывается архитектурно, но не обязан быть первой поставкой. Первая практичная поставка может быть локальной web-версией, которую клиент запускает из папки проекта через Codex/локальную среду. После стабилизации ядра эту же базу можно упаковать в desktop-приложение.

### Импорт/экспорт

- Excel/CSV: SheetJS или ExcelJS.
- PDF/печать КП: браузерная печать на первом этапе, позже генерация PDF.
- DOCX по необходимости: docx-js.

### AI-интеграции

AI/Codex должен быть важной частью поставки, но расчетная логика не должна полностью зависеть от чата. Продукт хранит данные структурно, а Codex помогает распознавать PDF, собирать прайсы, сопоставлять материалы и настраивать компанию.

Потенциальные задачи AI:

- сопоставление прайсов;
- анализ PDF;
- поиск похожих материалов;
- проверка пропущенных позиций;
- генерация текста КП.

## 3. Codex-powered поставка

Рабочая гипотеза поставки:

1. Подготавливается папка проекта.
2. Папка персонализируется под клиента: название, реквизиты, шаблоны, стартовые прайсы.
3. Клиент запускает проект локально.
4. Настройка проводится совместно через Codex: загружаются прайсы, PDF-планы, проверяются расчеты.
5. Итоговая рабочая версия остается у клиента локально.

Практические требования к такой поставке:

- понятный `README` для запуска;
- простой `start.bat`, который запускает приложение без ручного ввода команд;
- простой `start.command` / shell-скрипт для macOS;
- отдельный `backup.bat` и `backup.command` для резервной копии `client-data/`;
- сиды/демо-данные для первого старта;
- папка `client-data/` или аналог для прайсов, PDF и шаблонов;
- экспорт/импорт настроек компании;
- локальное хранение проектов;
- отсутствие обязательной серверной инфраструктуры на первом этапе.

## 4. Локальные данные и риск браузерной версии

Важный риск: обычная web-страница в браузере не может свободно читать и записывать файлы в папку проекта. Браузер ограничивает доступ к файловой системе из соображений безопасности.

Поэтому для поставки "папкой проекта" нужен один из двух рабочих путей:

### Путь A: локальный Node API

- `start.bat` на Windows и `start.command` на macOS запускают локальный сервер.
- React-приложение работает в браузере.
- Локальный API читает/пишет `client-data/`, SQLite, импорты и экспорты.
- Codex может помогать менять файлы проекта и оркестрировать настройку.

Плюс: быстрее стартовать разработку.

Минус: клиенту нужен Node.js или упакованный runtime.

Чтобы клиент ничего не устанавливал вручную, есть два варианта:

- поставлять инструкцию/скрипт, который проверяет Node.js и ставит зависимости автоматически;
- позже собрать standalone runtime или перейти к Tauri-сборке.

### Путь B: Tauri

- Приложение запускается как desktop.
- Tauri дает контролируемый доступ к файловой системе и SQLite.
- Можно упаковать в установщик/приложение.

Плюс: надежнее для клиента.

Минус: выше сложность старта, нужна Rust/Tauri-сборка.

Рекомендация: архитектуру сразу вести так, чтобы бизнес-логика была общей. Первый этап можно сделать через локальный Node API, а затем подключить Tauri как упаковку. Не завязывать расчетную модель на `localStorage`. Для клиента целевой UX запуска: двойной клик по скрипту или приложению, без ручных команд.

## 5. Почему не оставляем текущий CDN-прототип

Текущий проект полезен как UX-основа, но не подходит как финальная архитектура:

- нет модульной структуры;
- JSX подключается через CDN/Babel;
- нет типизации;
- нет нормальной модели данных;
- нет локальной БД;
- сложно тестировать и расширять.

Поэтому следующий этап - миграция в полноценную структуру `src/` на TypeScript.

## 7. Доменная модель

Минимальные сущности:

- Project.
- BuildingBlock: дом, этаж, гараж, баня, терраса, кровля.
- CanvasElement: wall, partition, room, opening, door, window, roofSegment, drain, aerator.
- Material.
- PriceList.
- PriceItem.
- Assembly / TechnologyPackage.
- Estimate.
- EstimateLine.
- Proposal.
- ProposalLine.
- CompanyProfile.

## 8. Источники ресерча

Аналоги и паттерны:

- TraceOff: PDF plans, scale, measurements, estimates, proposals.
- Pilrs: AI takeoff from 2D blueprints.
- Protakeoff: desktop takeoff, estimates, PDF/Excel/CSV export.
- Exayard: plan upload, AI takeoff, export.
- TakeCost: PDF/image plans, scope-based pricing, material and labor costs.
- BuildBid: review-first estimates with source-linked quantities and confidence flags.

Технические источники:

- React Konva / Konva.
- Mozilla PDF.js.
- Tauri.
- TanStack Table / Query.
- PGlite / local-first подходы.
